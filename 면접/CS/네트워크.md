## 데이터 전송 두 가지 방법
* circuit switching
    * 엣지간 연결 경로 유지
    * 비효율적
* Packet switching
    * 데이터를 패킷단위로 나눠 알아서 전송


## 네트워크
컴퓨터들이 통신망을 통해 그물처럼 연결된 체계를 의미합니다.
* 호스트, 스위치가 링크로 연결된 분산 시스템

* 호스트
    * 인터넷 끝에 연결된 종단장치
    * PC, 스마트폰 등
* 링크 
    * 통신 장치들 간에 패킷을 전달하는 유, 무선 매체
    * 구리선 WIFI 등
* 스위치
    * 다수의 통신 장치를 통신 링크로 연결하는 장치
    * 입력 링크로 받은 패킷을 출력 링크로 전달
    * 라우터 등

## Cast 종류
* UniCast 
    * 1 : 1 통신
    * 프레임에 자신의 MAC주소와 목적지의 MAC주소를 첨부하여 전송
    * 어떤 시스템이 유니캐스트 방식으로 데이터를 전송하게 되면 같은 네트워크에 있는 모든 시스템들은 그 MAC 주소를 받아 자신의 MAC 주소와 비교 후 같지 않다면 버리고 같다면 프레임을 받아서 처리
* Multicast
    * 1 : N 통신
    * 특정한 그룹에게만 프레임을 뿌린다
* Broadcast
    * 1 : all 통신
    * 로컬 네트워크의 모든 시스템에 프레임을 보내는 통신
    * 브로드캐스트용 주소가 미리 정해져 있다
        * 0.0.0.0
    * 같은 네트워크에 있는 모든 장비에게 전송

## 허브
여러 컴퓨터를 연결해서 네트워크를 만들어주는 장치

데이터 패킷을 받으면 연결된 모든 장치들에게 전송  
즉, 단순 분배를 하는 중계 장치

## 스위치
같은 LAN내에서 MAC 주소를 통해 목적지로 데이터 패킷을 전송하는 장치. 2계층에 속합니다.

연결된 장치들의 IP와 MAC 주소를 테이블 형태로 가지고, `원하는 목적지에 데이터 패킷을 전송하는 장치`

IP와 MAC 주소를 기반으로 필요한 장치에게만 전송

## 라우터
* 네트워크간 데이터 전송 위한 최적의 경로를 찾아 목적지까지 패킷을 전달하는 장비

* LAN을 연결해주는 장치
* 서브넷마스크가 다른 네트워크와 연결하기 위해 최적의 경로를 찾아 목적지까지 패킷을 전달하는 장비

## 공유기
네트워크의 컴퓨터들에게 IP 주소(사설)를 할당해주는 장치

## LAN(Local Area Network)
동일한 서브넷을 사용하는 근거리 통신망

근거리 통신망,  
네트워크 매체를 이용하여 집, 사무실, 학교 등 가까운 지역을 묶는 컴퓨터 네트워크

IP적으로
* 동일한 서브넷 마스크를 사용한다면 LAN

## IPv4
네트워크 상에서 데이터 교환하기 위한 3계층 프로토콜

데이터가 정확하게 전돨될 것을 보장하지 않는다
* 중복되거나 
* 순서를 보장하지 않음

* 정확하고 순차적인 전달은 상위 계층인 TCP가 담당

## ISP
인터넷 서비스 제공자와 그 조직이 운영하는 네트워크

## 프로토콜
원할한 통신을 위해 정보 교환 규칙을 상호합의를 통해 사전에 정의해놓은 것

## 클라이언트-서버 구조
* 클라이언트
    * 서버에게 응용 서비스 요청
    * 필요할 때만 작동
    * 동적 IP
* 서버
    * 클라이언트 서비스 요청에 응답
    * 항상 작동
    * 고정 IP 주소

## 소켓(Socket)
* 프로세스가 네트워크로 메시지를 송수신하는 통로 자료구조
* 트랜스포트 계층 상에서 구현
    * 애플리케이션 계층에서 보기에는 트랜스포트계층이 데이터 주고받는 것으로 보임
* 소켓 자료구조를 사용하여 통신 서비스를 제공하는 프로그램 인터페이스
* 소켓 API 가져다 응용 프로그램 구현

소켓 주소는 IP주소 + Port 번호로 이루어져 있다


## 패킷
전송 장치 간에 전달되는 정보 단위

데이터 링크 계층의 단위, 긴 메시지를 작은 패킷 단위로 나누어 전송한다

## Url을 입력했을 때 과정
도메인, ip, TCP 연결, 전송

1. 사용자가 URL 브라우저에 입력
2. 브라우저는 DNS를 통해 IP주소 획득
3. IP 주소를 통해 서버와 TCP 3 way handshake를 통해 연결
3. HTTP프로토콜을 통해 HTTP요청 메시지 생성
4. TCP/IP 연결을 통해 HTTP 요청 서버로 전송
5. 서버는 HTTP응답 메시지 회신
6. TCP/IP 연결을 통해 요청한 컴퓨터로 전송
7. 도착한 HTTP 응답 메시지 웹 페이지 데이터로 변환

## TCP, UDP
트랜스포트 계층의 프로토콜

* TCP
    * 연결지향적
    * 신뢰성 보장
    * 혼잡제어
        * 트래픽이 몰려 버퍼가 찬 경우 전송속도 조절합니다
    * 흐름제어
        * 몰라
    * 3wh, 4wh, 연결, 연결해제
* UDP
    * 비연결형 프로토콜
    * 연결을 유지하지 않기 때문에 패킷이 각각 다른 경로로 전송되게 됩니다
    * 신뢰성이 낮고, checksum을 통해 최소한의 오류만 검출합니다
    * 스트리밍 등에 사용됩니다

## 공인 IP vs 사설 IP
공인 IP
* ISP가 공급하는 유일한 주소입니다
* 공개되었기 때문에 인터넷에 연결된 다른 장비로부터 접근 가능합니다
* 인터넷 상에서 유일합니다

사설IP
* 가정이나 회사 등의 같은 네트워크에서 사용하는 IP입니다. 
* 공개되지않고 외부에서 접근이 불가능하고, 외부로 접근도 불가능합니다
* 라우터가 할당하게 됩니다
    * 이것을 NAT라 한다
* 하나의 네트워크 안에서 유일합니다
* IPv4의 IP부족문제를 해결하여 줍니다

* IPv4
    * 32비트
* IPv6
    * 128비트

## 게이트웨이
현재 사용자가 위치한 네트워크에서 다른 네트워크로 이동하기 위해 반드시 거쳐야 하는 거점

네트워크 상에서 서로 연결되려면 동일한 통신 프로토콜을 사용해야 합니다. 따라서 프로토콜이 다른 네트워크상의 컴퓨터와 통신하려면 두 프로토콜을 적절히 변환해주는 변환기가 필요한데 게이트웨이가 바로 이러한 변환기 역할을 합니다

게이트웨이(라우터)를 거쳐야만 외부 인터넷에 접속 가능

## 게이트웨이 주소
라우터(혹은 공유기)의 주소. 라우터는 네트워크와 네트워크를 연결해주는 장비로 공인 IP와 사설 IP 모두 가질 수 있다.

이때 라우터가 가지는 사설 IP가 게이트웨이

컴퓨터가 외부로 통신하고자 할 때, 라우터를 호출하는데 이때 라우터의 사설 IP를 호출함

참고로 라우터는 공유기 + 게이트웨이 + 등등 임

## ARP 프로토콜
같은 네트워크 대역에서 통신을 위해 IP주소를 통해 MAC주소를 알아오는 프로토콜, 3계층

IP 주소만 알고 MAC 주소를 모르더라도 ARP를 통해 통신이 가능

wildcard 주소를 목적지로 하여 브로드캐스트 하여 같은 네트워크 내의 모든 기기에 전송

패킷이 스위치로 가면 브로드캐스트인것 확인 후 스위치가 모든 기기로 전송
* 스위치는 2계층이기 때문에 2계층 까지만 디캡슐화
* 받은 기기는 3계층까지 까본 후 IP주소 확인하고 자신이면 ARP 응답

## 라우팅

## 라우팅 테이블
* 목적지까지의 거리와 경로를 명시하는 테이블
* 라우터는 라우팅 테이블을 보고 경로를 찾아갑니다

## 4계층 트랜스포트
* 송신자의 프로세스와 수신자의 프로세스를 연결하는 통신 서비스
* 신뢰성, 흐름 제어, 오류제어 등 여러 기능을 담당
* TCP/UDP

## TCP/IP 4계층
인터넷 통신을 4계층으로 나눈 것. 현재 인터넷은 TCP/IP 모델을 따른다
* 애플리케이션
    * FTP,HTTP,SSH
    * 응용 프로그램간 자료 송수신
* 트랜스포트
    * TCP,UDP
    * 호스트간 자료 송수신
* 인터넷
    * IP, ARP
    * 데이터 전송을 위한 논리적 주소 및 경로 지정
* 네트워크
    * MAC, LAN
    * 실제 데이터 전송

## OSI 7계층(웹 접근성 국제표준)
네트워크 통신을 7개의 계층으로 분리한 것입니다. 이를 통해 각 단계별로 의존성을 끊고 각자의 책임을 가지며 유지보수하고 개발하기 편해지는 장점이 있습니다.

캡슐화
* 상위 계층에서 받은 정보를 자신의 계층의 제어정보를 헤더화시켜 붙여 하위 계층으로 보냄

* 응용
    * 사용자가 네트워크 자원에 접근하는 방법을 제공
    * FTP, SMTP, HTTP
    * 응용 서비스를 수행하는 계층
    * 사용자에게 직접 보인다
* 프레젠테이션
    * 데이터 형식과 암호화 등을 담당한다
    * SSL, SSH, JPG
* 세션
    * 데이터가 통신하기 위한 논리적인 연결
    * TCP/IP 세션을 만들고 없애는 책임
* 트랜스포트
    * TCP, UDP
    * 종단간 서로 다른 호스트에서 동작하는 애플리케이션 프로세스간의 논리적 통신을 제공한다.
    데이터 전송 보장, 흐름을 제어하는 계층
    * 흐름제어, 혼잡제어 등을 담당
    * PORT
        * 프로세스간 통신을 위한 주소
        * 하나의 프로세스가 하나의 포트
* 네트워크 계층
    * 다른 네트워크 대역 간에 어떻게 데이터를 전달할지 제어하는 일을 담당
    * 데이터를 최적의 경로로 전달
    * 이것을 라우팅이라 한다
    * 경로를 선택하고, 주소를 정하고, 패킷을 전달
    * 대표적 장비로 라우터
    * 데이터 링크 계층에서는 MAC주소를 사용했다면, 네트워크 계층에서는 IP 주소를 사용
    * 주소를 부여(IP)하고 경로를 설정(라우팅)
    * IP 프로토콜은 비신뢰성, 순서나 신뢰성있는 전달을 보장하지 않는다
* 데이터링크
    * 같은 네트워크 대역의 두 장치간 신뢰성 있는 전송을 보장
    * 스위치나 브릿지
    * 랜 카드가 가지고 있는 MAC 주소라는 실제의 물리 주소를 통해 전송
        * MAC 주소는 16진수
* 물리 계층
    * 0, 1 비트 데이터를 케이블 등을 통해 전송

## TCP(Transmission Control Protocol)
* 연결형 서비스 신뢰성 보장
    * 데이터 전달, 중복 안됨, 전달 순서 등을 보장
* 가상회선 방식으로 패킷을 전달
* 3-way 논리적 연결, 4-way 해제
* 흐름제어
    * 송신 TCP가 지나치게 많은 데이터를 한꺼번에 송신함으로써 수신 TCP의 버퍼가 넘쳐 데이터 손실이 발생하는 문제를 방지하는 메카니즘
* 혼잡제어
    * 송신측의 데이터 전달과 네트워크 데이터 처리 속도의 차이를 해결하기 위한 기법

## UDP(User Datagram Protocol)
* 비연결지향 프로토콜
* 비연결형 서비스, 신뢰성이 보장되지 않는다
    * 순서도 보장하지 않는다
    * 연결을 설정하고 해제하는 과정이 없어 빠르다
* 그냥 IP에 포트 추가에 가까움, 체크섬 추가
* 1:1 or 1:N or N:N 통신
* 헨드쉐이킹과 혼잡, 흐름제어를 하지 않기 때문에 빠르다
* 스트리밍 등에 사용

## 포트 번호
* 특정 프로세스와 특정 프로세스가 통신을 하기 위해 사용
* 특정 프로그램 하나가 하나의 포트

## 캡슐화
프로토콜의 교환 정보는 *데이터 정보*와 *제어 정보*가 있다

## 3-way hand shake
TCP가 장치들간 논리적 연결을 수립하기 위한 과정

• 양쪽 모두 데이타를 전송할 준비가 되었다는 것을 보장하고, 실제로 데이타 전달이 시작하기전에 한쪽이 다른 쪽이 준비되었다는 것을 알수 있도록 한다.
• 양쪽 모두 상대편에 대한 초기 순차일련변호를 얻을 수 있도록 한다.

1. 클라이언트가 서버에게 접속을 요청하는 SYN 패킷 전송
    * syn flag 비트를 0
2. 서버는 SYN을 받아 요청을 수락한다는 ACK와 SYN을 클라이언트에게 전달
    * 
3. 클라이언트는 정상 수령했다면 최종 수락하는 ACK 반환
    * flag를 ack

이러면 연결이 된 것 

## 4-way hand shake
TCP연결을 종료하기 위한 프로토콜

1. 클라이언트가 연결을 종료하겠다는 FIN플래그 전송
2. 서버는 ACK을 모낸 다음 자신의 통신이 끝날때까지 wait
3. 서버가 통신이 끝났으면 연결이 종료되었다 클라이언트에게 FIN 전송
4. 클라이언트는 ACK 후 혹시 FIN보다 도착이 늦은 패킷이 있는지 일정기간 기다림(time-wait) 후 연결을 종료합니다

## NAT과 포트포워딩
* NAT
    * IP 패킷의 TCP/UDP 포트 숫자와 목적지 IP 주소 등을 재기록하면서 라우터를 통해 네트워크 트래픽을 주고받는 기술
    * 사설 네트워크에 속한 여러 호스트가 하나의 공인 IP 주소를 사용하여 인터넷에 접근하기 때문에 필요하다

* 포트포워딩
    * 특정 포트로 들어온 요청을 다른 특정 IP 특정 포트로 전송
    * 사설네트워크에 상주하는 호스트에 대한 서비스를 생성하기 위해 흔히 사용

## 쿠키, 세션
* 쿠키
    * http 통신은 statelss 하기 때문에 서버는 클라이언트에 대한 정보를 저장하지 않는다
    * 그래서 서버는 브라우저에 쿠키의 형태로 상태정보를 저장하게 한다
    * response 에 set cookie를 실어보내면 블라우저는 쿠키 저장소에 저장 후 서버에 요청시 같이 보낸다
* 세션
    * http 통신은 statelss 하기 때문에 서버는 클라이언트에 대한 정보를 저장하지 않는다
    * 따라서 세션이 필요함
    * 서버는 세션을 저장한 다음 쿠키로 세션 id를 브라우저에 전송한다
    * 브라우저는 서버에 요청 시 세션 id를 쿠키로 함께 보냄
    * 서버는 세션 id를 통해 브라우저에 대한 정보를 알 수 있다
* 쿠키는 브라우저에만 있어 앱에는 사용 불가
* 이런 경우는 토큰을 사용하게 된다

* 쿠키 vs 세션
    * 쿠키는 브라우저에 저장된다면
    * 세션은 서버의 세션 저장소에 저장되게 됩니다
    * 세션의 관리주체는 서버, 쿠키는 브라우저 입니다
    * 쿠키는 클라이언트 로컬에 저장되기 떄문에 탈취당하거나 변질될 위험이 있어
    * 보안면에서 세션이 더 우수합니다
    * 쿠키는 파일의 형태로 저장되어 브라우저를 종료해도 유지할 수 있고, 만료기간을 정할 수도 있지만
    * 세션은 브라우저가 종료되면 곧바로 삭제

## HTTPS
HTTP secure

HTTP에 암호화가 추가된 프로토콜입니다

443번 포트를 사용하며, 암호화를 지원합니다.

대칭키 암호화와 비대칭키 암호화를 모두 사용.

비대칭키 방식을 이용하여 세션키를 교환 후, 이후에는 대칭키를 사용하여 데이터 암호화하는 방식
1. 클라이언트가 서버로 최초 연결 시도
2. 서버는 인증서를 클라이언트에게 넘겨줌
3. 클라이언트는 인증서 검증 후 세션키 발급
4. 세션키를 서버의 공개키를 사용하여 암호화하여 전송
5. 서버는 개인키로 새션키 복호화
6. 대칭키 방식으로 통신

* 인증서는 검증된 기업에서 발급받는다
* 기업의 개인키로 암호화되어있기 때문에 기업의 공개키로 복호화된다면 해당 기업의 인증서이다
* 브라우저는 cA기업의 공개키를 미리 다운받아 갖고 있다

## OAuth2
사용자가 리소스 서버에 저장된 데이터에 대한 접근권한을  서드 파티 프로그램에게 위임하는 프로토콜

1. 애플리케이션이 인증 서버에 code 요청
2. 사용자 로그인
3. 인증 서버가 사전 정의된 redirect_url로 code 전달
4. 애플리케이션은 code를 이용하여 Access Token 요청
5. 해당 Access Token을 이용하여 resource server에 사용자 데이터 요청
6. resource server는 사용자 데이터 전달
